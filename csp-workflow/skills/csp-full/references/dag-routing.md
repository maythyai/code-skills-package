# CSP Full DAG 路由规则

## 输入复杂度评估

对用户的初始 prompt 进行以下维度的评分：

| 维度 | 低分 (0) | 中分 (1) | 高分 (2) |
|------|---------|---------|---------|
| 功能描述 | 无/模糊 | 有概述 | 详细功能列表 |
| 目标用户 | 未提及 | 笼统提及 | 明确定义 persona |
| 技术约束 | 无 | 部分 | 完整（栈、DB、部署） |
| 成功标准 | 无 | 定性描述 | 定量指标 |
| 时间线 | 无 | 模糊 | 明确 deadline |
| 依赖关系 | 无 | 提及 | 明确列出 |

**总分 0-4：极简/模糊 → P0 + P1 完整**
**总分 5-8：中等 → P0 轻量 + P1 标准**
**总分 9-12：详细 → 跳过 P0，P1 精简**

## 阶段跳过决策树

```
                    ┌──────────────────┐
                    │  检测已有产出物    │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
     PRD存在？         技术设计存在？     计划存在？
              │              │              │
         跳过 P1         跳过 P2        跳过 P3
              │              │              │
              └──────────────┼──────────────┘
                             │
                    ┌────────┴─────────┐
                    │  检测变更范围      │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
         简单 bug？      新功能？       重构？
              │              │              │
         跳 P0,P1,P2,P3   全流程        P2,P3 轻量
              │              │              │
              └──────────────┼──────────────┘
                             │
                    ┌────────┴─────────┐
                    │  检测技术栈        │
                    └────────┬─────────┘
                             │
         ┌───────────────────┼───────────────────┐
         │                   │                   │
     后端为主？           前端为主？           全栈？
         │                   │                   │
     激活后端 agents     激活前端 agents     激活全栈 agents
         │                   │                   │
         └───────────────────┼───────────────────┘
                             │
                    ┌────────┴─────────┐
                    │  执行 P4-P8       │
                    └──────────────────┘
```

## 分支场景

### 场景 A: 全新产品（最常见）
```
P0 → P1 → P2 → P3 → P4 → P5 → P6 → P7 → P8
```

### 场景 B: 已有 PRD 的新功能
```
跳过 P0,P1 → P2 → P3 → P4 → P5 → P6 → P7
```

### 场景 C: Bug 修复
```
跳过 P0,P1,P2,P3 → P4 → P5 → P6
（跳过 P7,P8 — 无需发布和运维）
```

### 场景 D: 技术重构
```
跳过 P0,P1 → P2(轻量) → P3 → P4 → P5 → P6
```

### 场景 E: MVP 迭代到 v1
```
P0(轻量) → P1(增量) → P2(增量) → P3 → P4 → P5 → P6 → P7 → P8
```

## Subagent 选择矩阵

根据任务类型和技术栈选择最优 subagent：

| 任务 | 技术栈 | Primary Agent | Secondary Agent |
|------|--------|--------------|-----------------|
| 后端 API | Python | `ecc:django-reviewer` | `ecc:fastapi-reviewer` |
| 后端 API | TypeScript | `ecc:typescript-reviewer` | `ecc:react-reviewer`(if SSR) |
| 后端 API | Go | `ecc:go-reviewer` | - |
| 后端 API | Java | `ecc:java-reviewer` | - |
| 后端 API | Rust | `ecc:rust-reviewer` | - |
| 前端 UI | React | `ecc:react-reviewer` + `ecc:react-build-resolver` | - |
| 前端 UI | Vue | `ecc:react-reviewer`(fallback) | - |
| 前端 UI | Flutter | `ecc:flutter-reviewer` | - |
| 数据库 | PostgreSQL | `ecc:database-reviewer` | - |
| 安全审查 | 任意 | `ecc:security-reviewer` | - |
| 测试 | 任意 | `ecc:tdd-guide` | 语言专项测试 reviewer |
| 架构设计 | 任意 | `ecc:architect` | - |
| 产品定义 | 任意 | 内置 PM 角色 | - |

## 模型选择策略

| 阶段 | 推荐模型 | 理由 |
|------|---------|------|
| P0 需求澄清 | Sonnet | 对话能力强，成本低 |
| P1 PRD | Opus | 需要深度推理和结构化 |
| P2 技术设计 | Opus | 架构决策需要高质量推理 |
| P3 规划 | Sonnet | 任务分解是机械性工作 |
| P4 执行 | 按复杂度：简单→Haiku, 标准→Sonnet, 复杂→Opus | 成本优化 |
| P5 QA | Sonnet | 需要理解错误模式 |
| P6 审查 | Opus | 多角度深度审查 |
| P7 发布 | Haiku | 机械性工作 |
| P8 运维 | Sonnet | 需要推理能力 |

## 门控条件详细说明

### P0 → P1 门控
- 开放问题 ≤ 3 个
- 核心问题已定义
- 至少 1 个目标用户 persona

### P1 → P2 门控
- PRD 包含 ≥ 1 个用户故事
- 每个用户故事有 ≥ 1 个 falsifiable 验收标准
- MVP 范围已明确（in-scope 和 out-of-scope）

### P2 → P3 门控
- 架构设计包含关键技术决策
- 数据模型已定义（如涉及数据存储）
- API 契约已定义（如涉及 API）
- 安全考量已评估

### P3 → P4 门控
- 任务列表使用 checkbox 格式
- 依赖关系已标注
- 并行策略已定义

### P4 → P5 门控
- 所有任务标记为 completed
- 代码编译/构建成功
- 无 lint 错误

### P5 → P6 门控
- 所有 CRITICAL 测试通过
- QA 循环 ≤ 5 轮
- 无重复 3 次的相同错误

### P6 → P7 门控
- 代码审查无 CRITICAL 问题
- PRD 对齐验证通过率 ≥ 90%
- 安全审查无 HIGH 漏洞

### P7 → P8 门控
- 部署成功
- RELEASE.md 已生成
- CHANGELOG.md 已更新
- Git tag 已创建
