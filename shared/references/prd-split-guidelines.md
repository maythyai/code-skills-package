# PRD Split Guidelines

> CSP PRD 拆分指南，帮助识别何时应该将大型 PRD 拆分为多个独立 change。

## 何时拆分

### 明确的拆分信号

| 信号 | 说明 | 示例 |
|------|------|------|
| 3+ 独立功能 | 功能之间没有强依赖 | 用户管理 + 订单系统 + 支付集成 |
| 多团队参与 | 不同团队负责不同模块 | 前端团队 + 后端团队 + 数据团队 |
| 阶段性交付 | 需要分阶段上线 | MVP → Beta → GA |
| 工作量 > 2 周 | 单个 change 预计超过 2 周 | 大型企业功能 |
| 高风险 | 失败会影响其他功能 | 核心系统重构 |

### 不建议拆分的情况

| 情况 | 原因 |
|------|------|
| 紧密耦合 | 功能之间强依赖，无法独立验证 |
| 原子性变更 | 必须一起上线才有意义 |
| 小工作量 | < 3 天的工作，拆分成本高于收益 |
| 单一模块 | 只涉及一个模块的变更 |

## 拆分原则

### 1. 独立性

每个拆分后的 change 应该：
- ✅ 可以独立设计
- ✅ 可以独立实现
- ✅ 可以独立测试
- ✅ 可以独立部署
- ✅ 可以独立回滚

### 2. 完整性

每个 change 应该是一个完整的交付单元：
- ✅ 有明确的业务价值
- ✅ 有清晰的验收标准
- ✅ 有完整的文档
- ✅ 有回滚方案

### 3. 合理的粒度

避免过度拆分：
- ❌ 拆分成太小的 change（< 1 天工作量）
- ❌ 拆分后仍需大量协调
- ❌ 拆分破坏了功能的完整性

## 拆分方法

### Step 1: 识别功能边界

分析 PRD，识别独立的功能模块：

```markdown
# PRD: 电商平台

## 功能列表
1. 用户注册和登录
2. 商品浏览和搜索
3. 购物车管理
4. 订单创建和支付
5. 物流跟踪
6. 评价系统
```

### Step 2: 分析依赖关系

绘制功能依赖图：

```
用户注册 ──► 购物车 ──► 订单支付 ──► 物流跟踪
                │
                └──► 评价系统

商品浏览 ──► 购物车
```

### Step 3: 确定拆分方案

基于依赖关系，确定独立的 change：

```markdown
## 拆分方案

### Change 1: 用户系统
- 用户注册
- 用户登录
- 用户信息管理

### Change 2: 商品系统
- 商品浏览
- 商品搜索
- 商品详情

### Change 3: 购物流程
- 购物车管理
- 订单创建
- 支付集成

### Change 4: 售后服务
- 物流跟踪
- 评价系统
```

### Step 4: 确定执行顺序

基于依赖关系，确定执行顺序：

```
Change 1 (用户系统)
    ↓
Change 2 (商品系统)
    ↓
Change 3 (购物流程) - 依赖 1 和 2
    ↓
Change 4 (售后服务) - 依赖 3
```

## 拆分后的工作流

### 1. 创建多个 change

```bash
# 为每个拆分创建独立的 change
./shared/scripts/csp-state.sh init full --name "user-system"
./shared/scripts/csp-state.sh init full --name "product-system"
./shared/scripts/csp-state.sh init full --name "shopping-flow"
./shared/scripts/csp-state.sh init full --name "after-sales"
```

### 2. 记录依赖关系

在每个 change 的 proposal 中记录依赖：

```markdown
# Proposal: shopping-flow

## Dependencies
- Depends on: user-system (Change 1)
- Depends on: product-system (Change 2)

## Why
需要用户系统和商品系统提供基础能力
```

### 3. 按顺序执行

```bash
# 先执行无依赖的 change
./shared/scripts/csp-state.sh set current_change user-system
# ... 完成 user-system ...

./shared/scripts/csp-state.sh set current_change product-system
# ... 完成 product-system ...

# 再执行有依赖的 change
./shared/scripts/csp-state.sh set current_change shopping-flow
# ... 完成 shopping-flow ...
```

## 拆分检查清单

### 拆分前检查

- [ ] PRD 是否包含 3+ 个独立功能？
- [ ] 是否有多个团队参与？
- [ ] 预计工作量是否 > 2 周？
- [ ] 是否需要分阶段交付？
- [ ] 失败风险是否会影响其他功能？

### 拆分后检查

每个 change 是否满足：
- [ ] 可以独立设计？
- [ ] 可以独立实现？
- [ ] 可以独立测试？
- [ ] 可以独立部署？
- [ ] 可以独立回滚？
- [ ] 有明确的业务价值？
- [ ] 有清晰的验收标准？

### 执行顺序检查

- [ ] 依赖关系是否清晰？
- [ ] 执行顺序是否合理？
- [ ] 是否有循环依赖？（如果有，需要重新拆分）
- [ ] 每个 change 的 proposal 是否记录了依赖？

## 常见拆分模式

### 模式 1: 按功能模块拆分

```
大型功能
├── 模块 A (独立功能)
├── 模块 B (独立功能)
└── 模块 C (独立功能)
```

**适用**: 功能之间耦合度低

### 模式 2: 按用户旅程拆分

```
完整流程
├── 阶段 1: 用户入口
├── 阶段 2: 核心流程
└── 阶段 3: 结果展示
```

**适用**: 线性业务流程

### 模式 3: 按技术层次拆分

```
完整系统
├── 数据层 (数据库、API)
├── 业务层 (服务、逻辑)
└── 展示层 (UI、交互)
```

**适用**: 技术栈分层明确

### 模式 4: 按交付阶段拆分

```
完整产品
├── MVP (最小可行产品)
├── Beta (测试版本)
└── GA (正式发布)
```

**适用**: 需要快速验证的产品

## 拆分决策树

```
PRD 包含 3+ 个独立功能？
├─ 是 → 有强依赖吗？
│       ├─ 否 → ✅ 按功能拆分
│       └─ 是 → 可以解耦吗？
│               ├─ 是 → ✅ 解耦后拆分
│               └─ 否 → 保持单一 change
└─ 否 → 预计工作量 > 2 周？
        ├─ 是 → 有阶段性交付需求吗？
        │       ├─ 是 → ✅ 按阶段拆分
        │       └─ 否 → 保持单一 change
        └─ 否 → 保持单一 change
```

## 与 Comet 的差异

| 特性 | Comet | CSP |
|------|-------|-----|
| 触发条件 | 大型 PRD | 3+ 独立功能或 > 2 周 |
| 拆分方法 | 手动分析 | 手动分析 + 决策树 |
| 依赖记录 | `.comet.yaml` | `workflow-state.yaml` |
| 执行顺序 | 用户决定 | 用户决定 + 建议 |

## 最佳实践

1. **尽早拆分** - 在 open 阶段就确定拆分方案
2. **记录依赖** - 每个 change 的 proposal 中明确记录依赖
3. **按顺序执行** - 先执行无依赖的 change
4. **保持独立性** - 避免拆分后的 change 之间产生新的依赖
5. **定期回顾** - 每个 change 完成后回顾拆分是否合理
